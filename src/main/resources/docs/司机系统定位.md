# 一、系统定位

这是一个 **企业货运运力管理平台**，核心是管理：

- 企业
- 车队
- 司机
- 车辆
- 证件
- 审批
- 任务
- 接单执行

系统有三个主要端：

- **司机端**：注册、上传证件、查看任务、申请换绑、接单、上报执行节点
- **管理端**：审核资料、分配车队、分配车辆、审批换绑、发布任务、异常处理
- **调度端**：面向运力调度，给车队/司机派单，处理执行中的异常

------

# 二、核心组织结构

## 1. 组织层级

我把系统组织关系定义为：

**企业 -> 车队 -> 司机 / 车辆**

也就是说：

- 一个企业下有多个车队
- 一个车队下有多名司机
- 一个车队下有多辆车

车队是企业下面的运力组织单元。

------

## 2. 为什么要有车队

车队主要解决 4 件事：

- 运力分组管理
- 调度隔离
- 权限隔离
- 首次分车、后续换绑的组织归属

所以系统不是企业直接平铺管理司机和车辆，而是通过车队管理。

------

# 三、核心业务对象

------

## 1. 司机

司机是任务执行主体，但司机不是注册完、证件过审就能直接接单。

司机要真正具备接单资格，需要同时满足：

- 证件审核通过
- 已分配车队
- 已分配合规车辆
- 当前状态正常
- 当前无风控限制

------

## 2. 车辆

车辆是任务执行载体，也有自己的证件、审核状态和所属车队。

车辆不能直接被任意司机使用，而是要么：

- 与司机建立当前有效绑定关系
- 要么在任务维度被临时指定执行

------

## 3. 车队

车队是运力管理单元。

车队下面通常有：

- 司机
- 车辆
- 车队长
- 调度员
- 本车队任务池

任务可以发给某个车队，而不是直接发给全企业。

------

## 4. 绑定关系

司机和车辆之间不直接靠一个字段互相挂 ID，而是通过 **绑定关系表** 管理。

原因是绑定关系有：

- 历史
- 状态
- 审批过程
- 生效/失效时间
- 同队/跨队信息

------

# 四、首次入驻与首绑逻辑

这块是很容易被问的，也是必须讲顺的。

## 1. 司机注册

司机张三先在司机端注册，并上传：

- 身份证
- 驾驶证
- 从业资格证

## 2. 证件审核

管理端审核通过后，司机并不会立刻可接单，而是先进入：

- `CERTIFIED_WAIT_TEAM`

意思是：已认证，但还没分配车队。

## 3. 分配车队

由企业管理员把张三分到某个车队，比如“成都冷链一队”。

分完后状态变为：

- `CERTIFIED_WAIT_VEHICLE`

意思是：已认证、已入队，但还没分车。

## 4. 首次分车

首次分车由企业或车队长主导，而不是司机自己盲选。

因为新司机通常不知道：

- 哪辆车空闲
- 哪辆车适合自己
- 哪辆车证件齐全
- 企业今天怎么排班

因此首次分车应由车队在本队车辆池中分配。

## 5. 激活

只有当司机满足：

- 已认证
- 已入队
- 已绑定合规车辆

状态才进入：

- `ACTIVE`

这时司机才真正具备接单资格。

------

# 五、换绑车辆逻辑

## 1. 首绑和换绑要区分

这是统一口径里非常重要的一点。

### 首绑

由企业/车队主导分配。

### 换绑

司机可以申请，企业也可以发起调整。

------

## 2. 换绑的发起方

系统里支持两种发起方式：

### 司机发起

适用于：

- 原车故障
- 日常申请换车
- 企业开放候选车辆池后司机申请

### 企业发起

适用于：

- 调度调整
- 车队重新排班
- 运力重新配置

------

## 3. 换绑不直接改正式关系

不管谁发起，都不能直接改正式绑定关系。

标准流程是：

1. 生成换绑申请
2. 生成审批单
3. 审批通过后，事务性切换正式绑定关系

------

## 4. 同车队与跨车队

### 同车队换绑

是常规场景，审批可相对简单。

### 跨车队换绑

属于更敏感场景，会影响：

- 司机归属
- 车队运力
- 任务池
- 权限边界

所以默认不能直接换，通常要走更严格审批。

------

# 六、任务发布与接单逻辑

------

## 1. 任务发布范围

任务支持三种范围：

- **指定司机**
- **指定车队**
- **企业公开任务池**

也就是任务表里会有：

- `dispatch_scope = DRIVER / TEAM / ENTERPRISE`

------

## 2. 指派单和车辆关系

企业在指派任务时，可以指定本次任务的执行车辆。

但这里要注意：

**任务执行车辆 != 正式绑定车辆**

也就是说：

- 企业可以在任务里指定一辆车给司机执行本次任务
- 但这不代表修改司机的长期主绑定关系

正式绑定是主数据，任务车辆是单次执行快照，二者分开。

------

## 3. 司机接单前校验

司机接单时，系统会聚合判断：

- 司机状态是否正常
- 司机证件是否有效
- 当前绑定车辆是否存在
- 当前绑定车辆证件是否有效
- 当前是否有进行中任务
- 当前任务是否仍可接
- 若是指派单，是否是被指派本人
- 若是车队任务，是否属于目标车队

只有全部通过，才允许接单。

------

## 4. 抢单并发控制

多个司机同时抢单时，最终一致性靠数据库状态条件更新保证。

也就是说不是靠内存锁，也不是只靠 Redis 锁，而是靠类似：

- 任务当前必须仍处于 `PUBLISHED`
- 谁先更新成功谁接到单

Redis 锁最多做削峰辅助，不做最终兜底。

------

# 七、证件逻辑

------

## 1. 证件分类

### 司机证件

- 身份证
- 驾驶证
- 从业资格证

### 车辆证件

- 行驶证
- 道路运输证
- 保险单等

------

## 2. 证件管理方式

证件不是简单存图片，而是存结构化数据，包括：

- 证件类型
- 证件号
- 主体姓名
- 有效期
- 审核状态
- 版本号
- 是否当前版本
- OCR 结果
- 驳回原因

------

## 3. 证件更新策略

证件重新上传时，不覆盖旧记录，而是新增新版本。

原因：

- 保留历史
- 保留审核留痕
- 支持追溯
- 防止直接覆盖造成审计断层

------

## 4. 系统怎么判断证件过期

不是靠人工改“过期状态”，而是基于：

- 审核状态
- 有效期开始结束时间
- 当前版本标记
- 作废标记

实时判断证件当前是否具备业务效力。

也就是说，真正判断的是：

**证件当前是否可用**
 而不是只看一个静态“过期”字段。

------

# 八、证件过期与任务执行

这个点前面我们已经修正成强合规版口径。

## 1. 接单前

只要核心证件过期，就直接禁止接单。

------

## 2. 执行中

如果任务执行到一半，核心合规证件过期，比如：

- 驾驶证
- 行驶证
- 道路运输证

系统不能默认继续正常执行，而应该进入风险处置流程。

### 系统动作

- 任务转为 `RISK_HOLD`
- 禁止继续正常节点流转
- 生成异常工单
- 通知调度、管理端、风控人员
- 后续由人工决定：
  - 换司机
  - 换车
  - 转派
  - 中断关闭

也就是说：

**合规硬门槛证件过期，当前任务也不能当作正常任务继续跑。**

# 九、状态流设计

这是面试里必须讲清的部分。

------

## 1. 司机状态流

我建议统一为：

- `INIT`：刚注册
- `WAIT_AUDIT`：资料待审核
- `CERTIFIED_WAIT_TEAM`：审核通过，待分配车队
- `CERTIFIED_WAIT_VEHICLE`：已入队，待分配车辆
- `ACTIVE`：已认证、已入队、已绑定车，可接单
- `FROZEN`：冻结
- `DISABLED`：停用

### 流程

```
INIT -> WAIT_AUDIT -> CERTIFIED_WAIT_TEAM -> CERTIFIED_WAIT_VEHICLE -> ACTIVE
```

异常分支：

- 审核失败可回退补资料
- 风控或违规可进 `FROZEN`
- 离职、注销可进 `DISABLED`

------

## 2. 车辆状态流

- `WAIT_AUDIT`
- `ACTIVE`
- `REJECTED`
- `FROZEN`
- `DISABLED`

------

## 3. 证件状态流

证件本身主要是审核状态：

- `WAIT_AUDIT`
- `APPROVED`
- `REJECTED`

“是否过期”通常是由有效期动态计算出的业务状态，而不是单独硬编码成唯一判断依据。

------

## 4. 绑定关系状态流

- `PENDING_APPROVAL`
- `ACTIVE`
- `REJECTED`
- `UNBOUND`
- `EXPIRED`

### 流程

首次绑定或换绑申请后：

```
PENDING_APPROVAL -> ACTIVE
```

驳回：

```
PENDING_APPROVAL -> REJECTED
```

换绑成功时：

旧绑定：`ACTIVE -> UNBOUND`
 新绑定：`PENDING_APPROVAL -> ACTIVE`

------

## 5. 审批单状态流

- `PENDING`
- `APPROVED`
- `REJECTED`
- `CANCELLED`

------

## 6. 任务状态流

主链路：

- `DRAFT`
- `PUBLISHED`
- `ACCEPTED`
- `IN_TRANSIT`
- `ARRIVED`
- `COMPLETED`

异常链路：

- `CANCELLED`
- `CLOSED`
- `RISK_HOLD`
- `TRANSFERRED`

### 主流程

```
DRAFT -> PUBLISHED -> ACCEPTED -> IN_TRANSIT -> ARRIVED -> COMPLETED
```

### 异常流程

执行中出现严重合规问题：

```
IN_TRANSIT -> RISK_HOLD
```

后续可能：

- `RISK_HOLD -> TRANSFERRED`
- `RISK_HOLD -> CANCELLED`
- `RISK_HOLD -> CLOSED`

------

# 十、审批模型

审批是这套系统的重要中枢。

## 1. 哪些业务要审批

- 司机证件审核
- 车辆证件审核
- 首次绑定（看企业规则）
- 换绑车辆
- 跨车队调拨
- 特殊异常恢复

------

## 2. 审批单为什么要和主表分开

因为审批中的数据是申请态，不是正式态。

如果不分开，会导致：

- 未批准的数据污染正式数据
- 驳回后难恢复
- 审计链路不清晰

所以审批单表和正式业务表分离是必须的。

------

## 3. 审批单为什么要存快照

因为审批页面需要看到“提交当时”的内容，而不能依赖后续已变化的主表数据。

------

# 十一、权限模型

------

## 1. 平台管理员

可看所有企业。

## 2. 企业管理员

只看本企业数据。

## 3. 车队长

只看本车队司机、车辆、任务、审批。

## 4. 调度员

可以是企业级或车队级。

## 5. 司机

只看自己和自己能处理的任务。

------

## 2. 数据权限核心原则

权限控制不能只靠前端隐藏按钮，后端必须做：

- 角色校验
- 企业归属校验
- 车队归属校验

例如车队长不能审批别的车队司机的数据。

# 十三、一个完整业务例子

你可以直接用这段讲系统，非常自然。

> 比如一个新司机张三先在司机端注册，上传身份证、驾驶证和从业资格证，审核通过后不会直接可接单，而是先进入待分配车队状态。企业把他分到“成都冷链一队”后，再由车队长从本队可用车辆池里给他分配首辆车。分车成功后，张三才正式进入可用状态。
>  后续如果原车故障，张三可以申请换绑本车队内其他车辆；如果涉及跨车队换车，则需要更严格审批。
>  调度发布任务时，可以选择指派给某个司机、某个车队，或者进入全企业任务池。司机接单前，系统会同时校验司机本人证件、当前绑定车辆证件、任务状态、车队归属和当前空闲状态。
>  如果任务执行中出现核心证件过期，任务不会继续按正常链路流转，而是进入风险挂起状态，由调度人工决定换车、换司机、转派或中断关闭。

这段基本能把整套系统讲圆。

# 十四、最后给你一句总括

你可以把整套系统总结为：

> 这套系统的核心不是单独管理司机或车辆，而是围绕“运力组织、资格控制、绑定关系、任务执行和审批风控”来设计。企业通过车队组织运力，司机和车辆通过绑定关系形成执行单元，证件和审批决定资格边界，任务系统承接调度与履约，而状态流和异常流共同保证业务可控和合规。
